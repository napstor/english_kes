# English KES Product Specification

Версия: 2026-04-28  
Статус: рабочее ТЗ для разработки полного продукта  
Источник методологии: книга `content/source/givental-kak-eto-skazat-po-angliyski-2013.md` и PDF-оригинал.

## 1. Суть проекта

English KES — это web-тренажер разговорного английского для русскоязычного взрослого пользователя, который начинает говорить по-английски через системную отработку артикуляционно-грамматических паттернов книги.

Главная цель продукта: пользователь, прошедший все уроки полностью и добросовестно, должен уверенно, спонтанно и достаточно быстро говорить простыми английскими фразами без постоянного внутреннего перевода и без паралича перед построением предложения.

Продукт не является обычным учебником, словарем или тестовой платформой. Это тренажер автоматизации речи. Пользователь должен многократно:

1. Понять грамматическую формулу.
2. Услышать лексику и эталонные фразы.
3. Собрать английскую фразу из русской мысли.
4. Получить точную проверку ошибок.
5. Исправить ошибку самостоятельно, не получая готовый ответ слишком рано.
6. Проговорить правильный вариант вслух.
7. Сравнить свою запись с эталоном.
8. Получить оценку произношения, ритма, ударения и интонации.
9. Вернуться к слабым местам в повторении.

Ключевая продуктовая метрика: не “пользователь прошел экран”, а “пользователь может произнести нужную конструкцию без подсказки, в темпе и с понятным произношением”.

## 2. Целевая аудитория

Основной пользователь:

- взрослый русскоязычный ученик;
- уровень английского: A0-B1;
- в голове сильная русская логика построения фраз;
- хочет не просто “знать правила”, а начать говорить;
- готов делать много повторений, если интерфейс ясно ведет его по процессу;
- не хочет академического шума, длинных объяснений и лишней теории.

Администратор:

- создает учеников по логину и паролю;
- видит базовый прогресс;
- в дальнейшем может назначать уроки, смотреть слабые места и записи.

## 3. Принципы продукта

### 3.1. Методология важнее экрана

Каждый экран должен обслуживать конкретный учебный шаг. Нельзя превращать продукт в набор карточек, где пользователь нажимает “Дальше” без реальной отработки.

### 3.2. Английский эталон не должен быть постоянно открыт

В заданиях на говорение и перевод пользователь сначала видит русскую мысль и должен сам произнести или написать английский вариант. Английский эталон скрыт и открывается только по клику “Показать эталон”. Это нужно, чтобы тренировать спонтанное извлечение фразы, а не чтение с листа.

### 3.3. Сократовская проверка

Когда пользователь ошибается в письменных заданиях или самостоятельных 10-20 предложениях, AI не должен сразу давать готовый ответ. Сначала он:

- кратко называет тип ошибки;
- дает минимальный блок теории;
- задает вопрос или подсказку, которая помогает пользователю исправить самому;
- подсвечивает строку/место ошибки;
- предлагает повторить попытку.

Готовый вариант можно показывать только после второй/третьей неудачной попытки или по явному действию пользователя.

### 3.4. Современный английский выше буквального копирования книги

Книга используется как методологическая база, но в ней есть редакторские ошибки, OCR-артефакты и устаревшие/неестественные фразы. Приложение должно преподавать современный естественный английский.

Примеры:

- `week-days` -> `weekdays`;
- `week-ends` -> `weekends`;
- `weeek-days` -> исправить по смыслу;
- `to do the room` -> `to clean the room` / `to tidy the room`;
- `to come in time` для “в последнюю минуту” -> `to come at the last minute`;
- `to beautify oneself` для “навести макияж” -> `to put on makeup` / `to do one's makeup`;
- `supper` не основной вариант для beginner-речи, основной — `dinner`, если контекст не требует иного.

Все такие правила должны фиксироваться в `content/editorial/rules.json` и проверяться через `npm run audit:editorial`.

## 4. Конечный результат для пользователя

После полного прохождения всех уроков пользователь должен:

- быстро строить утверждения, вопросы и отрицания;
- автоматически выбирать вспомогательные глаголы;
- не забывать окончания и базовую форму глагола;
- правильно ставить слова частоты;
- понимать и использовать лексику уроков в речи;
- воспроизводить speech drills без чтения;
- говорить достаточно громко, внятно и ритмично;
- слышать разницу между своей речью и эталоном;
- иметь список слабых мест для повторения;
- возвращаться к предыдущим урокам и закреплять их по алгоритму книги.

## 5. Методологический флоу урока

Каждый урок должен строиться по алгоритму книги:

1. Разобраться в теоретическом материале.
2. Поработать с “Техникой речи 1”.
3. Используя новые слова и выражения, самостоятельно составить 10-20 предложений по образцу “Техники речи 1”.
4. Повторить новую лексику с диктором.
5. Поработать с “Техникой речи 2”.
6. Быстро проговорить “Технику речи 2” двух предыдущих уроков.
7. Прослушать треки Speech Drilling 2 трех предыдущих уроков.

Продукт должен превратить этот алгоритм в маршрут урока, а не просто показать пункты.

## 6. Типы шагов урока

### 6.1. Теория

Цель: быстро понять формулу и слабые места.

Экран должен содержать:

- короткое название навыка;
- главную идею урока;
- формулу;
- 2-4 карточки с правилами;
- 2-4 примера;
- блок “частые ошибки русскоязычных”;
- кнопку “Понял, продолжить”.

Теория должна быть написана ясным русским языком, без академического перегруза. Для английского интерфейса должны быть английские тексты, но методология остается такой же.

### 6.2. Лексика и идиомы

Цель: подготовить словарь, который будет использоваться в упражнениях.

Экран должен содержать:

- список слов и выражений;
- русский перевод;
- английский вариант;
- кнопку прослушивания каждого элемента;
- режим “прослушать все”;
- режим “повторить вслух”;
- отметку прослушанного/отработанного.

Все английские элементы должны озвучиваться через ElevenLabs и кешироваться, чтобы не дергать API повторно.

### 6.3. Техника речи 1

Цель: автоматизировать базовую грамматическую конструкцию.

Для Present Simple пример формата:

- русская фраза: “Летом ты играешь в теннис.”
- пользователь пишет три английские формы подряд:
  - утверждение;
  - вопрос;
  - отрицание.

Проверка:

- сначала локальный deterministic scoring;
- затем, если есть ошибка или неоднозначность, OpenAI coach-check;
- подсветка слов и строк;
- краткая грамматическая справка;
- сократовская подсказка;
- повторная попытка.

### 6.4. Самостоятельные 10-20 предложений

Это обязательный шаг по методологии книги. Его нельзя пропускать.

Флоу:

1. Пользователь видит задание: “Используя лексику урока, составь 10-20 предложений по образцу Техники речи 1”.
2. Интерфейс дает 10 строк по умолчанию и кнопку “+ еще строка” до 20.
3. В каждой строке пользователь пишет полный паттерн, например:
   - `He always forgets to charge his phone.`
   - `Does he always forget to charge his phone?`
   - `He doesn't always forget to charge his phone.`
4. OpenAI проверяет:
   - разнообразие предложений;
   - использование лексики урока;
   - соответствие грамматическому паттерну;
   - типовые ошибки;
   - повторение одного и того же примера;
   - грубые spelling/word order ошибки.
5. Ошибочные строки подсвечиваются.
6. AI дает не готовые ответы, а краткую теорию и вопросы для самостоятельного исправления.
7. После исправления пользователь повторно отправляет работу.

Критерий завершения: минимум 10 валидных предложений, score >= 80 или ручное подтверждение админа в будущем.

### 6.5. Техника речи 2

Цель: использовать лексику/идиомы и формулу в более разнообразных фразах.

Экран похож на “Техника речи 1”, но фразы могут быть менее шаблонными. При проверке AI должен учитывать допустимые естественные варианты, а не требовать буквального совпадения с ключом книги.

### 6.6. Говорение

Цель: перевести внутреннюю письменную конструкцию в устную автоматическую речь.

Главный модуль speaking-экрана должен быть единым тренировочным блоком:

- русская фраза крупно и ясно;
- инструкция отдельно от фразы;
- скрытый английский эталон;
- кнопка записи;
- индикатор записи;
- результат распознавания;
- кнопка прослушать свою запись;
- кнопки эталона: медленно / обычный темп;
- оценка: смысл, грамматика, произношение;
- подсветка расхождений;
- интонационные подсказки стрелками/разметкой;
- рекомендация “что повторить сейчас”;
- кнопка “Повторить”;
- кнопка “Завершить”, доступная только после достаточной попытки.

Пользователь должен иметь возможность сравнить:

1. Что он должен был сказать.
2. Что он реально сказал.
3. Как это звучит у диктора.
4. Какие слова/звуки/ритм надо поправить.

### 6.7. Повторение предыдущих уроков

Система должна автоматически добавлять в маршрут:

- “Техника речи 2” двух предыдущих уроков;
- Speech Drilling 2 трех предыдущих уроков.

Нужен отдельный раздел “Повторение”, где пользователь видит:

- что просрочено;
- что слабое;
- что надо повторить сегодня;
- какие конструкции уже автоматизированы.

## 7. AI-функции

### 7.1. Проверка письменного ответа

Endpoint: `app/api/coach/check-answer/route.ts`

AI-роль:

> Ты коуч-учитель английского языка для взрослых русскоязычных, которые осваивают английский с нуля и мыслят русской логикой. Объясняй максимально ясно, кратко и прикладно. Не перегружай. Покажи суть ошибки и помоги исправить.

Требования к ответу:

- JSON-структура;
- score 0-100;
- статус: correct / acceptable / needs_fix;
- краткая причина;
- highlighted tokens;
- лучший вариант;
- correction pairs;
- краткая грамматическая справка;
- micro-drill для закрепления.

### 7.2. Проверка самостоятельных 10-20 предложений

Endpoint: `app/api/coach/composition/route.ts`

AI должен:

- проверять каждую строку отдельно;
- возвращать номера строк с проблемами;
- не раскрывать готовые ответы сразу;
- давать сократовские вопросы;
- отличать грамматическую ошибку от просто альтернативного естественного варианта;
- поощрять разнообразие лексики;
- не принимать 10 одинаковых предложений.

### 7.3. Распознавание речи

Endpoint: `app/api/audio/upload/route.ts`

Запись пользователя:

- сохраняется в Vercel Blob как private object;
- распознается через OpenAI STT;
- сверяется с target/accepted answers;
- получает pronunciation review.

### 7.4. Pronunciation review

AI должен оценивать не только текстовое совпадение, но и то, как речь воспринимается носителем:

- понятность;
- грубые ошибки произношения;
- ритм;
- ударение;
- интонация;
- паузы;
- звучит ли фраза естественно.

Важно: система не должна требовать идеальный акцент. Цель — понятная и не режущая слух речь.

Вывод:

- score 0-100;
- короткий общий вывод;
- 1-3 конкретных замечания;
- разметка фразы стрелками:
  - `↑` rising intonation;
  - `↓` falling intonation;
  - `•` stress;
  - `/` pause;
- что повторить прямо сейчас.

### 7.5. TTS

Endpoint: `app/api/tts/route.ts`

Требования:

- ElevenLabs v3;
- британские голоса;
- голос может выбираться случайно из whitelist;
- для одной фразы желательно сохранять стабильный voice/variant;
- медленная и обычная версии для speech drills;
- кеширование в Vercel Blob;
- нельзя повторно генерировать один и тот же простой audio asset, если он уже есть.

## 8. UX и визуальная система

Общий стиль: современный, спокойный, премиальный тренажер речи. Можно вдохновляться Duolingo, но не копировать. Интерфейс должен быть взрослым, сфокусированным и типографически сильным.

### 8.1. Layout

Desktop:

- левый sidebar: навигация, разделы, прогресс;
- центральная рабочая зона: один главный учебный фокус;
- правая панель не должна постоянно шуметь;
- админские функции не должны перекрывать учебный экран.

Mobile:

- mobile-first fallback обязателен;
- sidebar превращается в нижнюю или выезжающую навигацию;
- один экран — одно действие;
- запись речи должна быть доступна большим thumb-friendly контролом.

### 8.2. Типографика

Нельзя сливать инструкцию и учебную фразу в один огромный заголовок.

Правильно:

- label: “Задание”;
- instruction: средний размер, нейтральный цвет;
- Russian prompt: отдельный крупный блок;
- hidden English target: отдельный reveal-блок;
- feedback: ниже, в своей зоне.

Русская фраза должна быть главным визуальным объектом в speaking/translation упражнениях.

### 8.3. Цвета

Палитра должна быть не однотонной зеленой кашей. Зеленый — основной action/status цвет, но нужны:

- нейтральный фон;
- темный текст;
- приглушенный sage/gray для вторичных блоков;
- blue для theory/help;
- amber для warnings/acceptable;
- red только для реальных ошибок;
- достаточно контраста.

### 8.4. Cards

Не делать карточки внутри карточек без необходимости. Карточки использовать для:

- отдельных учебных блоков;
- feedback;
- theory cards;
- admin panels/modals.

Главный экран должен быть чистым, не перегруженным.

### 8.5. Admin UX

Админка нужна, но она вторична.

Минимально:

- создать ученика: login/password;
- список пользователей;
- сменить пароль;
- удалить/деактивировать пользователя в будущем.

Не нужно показывать пользователю служебные статусы вроде “прогресс пока хранится локально”, если они не помогают выполнить задачу. Технические статусы должны жить в dev/admin diagnostics, а не в основном UI.

## 9. Авторизация и роли

Текущий подход: простая серверная авторизация по логину/паролю, без email.

Роли:

- `admin`;
- `student`.

Требования:

- admin создается через env/bootstrap;
- admin создает student accounts;
- password hash хранится в Postgres;
- session cookie httpOnly;
- студент видит только свой прогресс;
- админ видит список учеников.

Платежей нет. Продукт бесплатный на текущем этапе.

## 10. Хранение данных

### 10.1. Neon Postgres

Использовать для:

- users;
- sessions;
- lesson progress;
- attempts;
- composition submissions;
- weak spots;
- audio metadata;
- generated audio cache metadata в будущем.

Минимальные таблицы:

- `users`;
- `sessions`;
- `progress`;
- `attempts`;
- `composition_attempts`;
- `speech_attempts`;
- `weak_spots`;
- `lesson_assets`.

### 10.2. Vercel Blob

Использовать для:

- записей пользователя;
- сгенерированных ElevenLabs audio assets;
- возможно, подготовленных lesson audio bundles.

Blob должен быть private для пользовательских записей. Для TTS можно private + signed access через API.

## 11. Content pipeline

Источник книги уже распарсен в:

`content/source/givental-kak-eto-skazat-po-angliyski-2013.md`

Редакторские правила:

`content/editorial/rules.json`

Отчет:

`content/editorial-report.md`

Команда:

```bash
npm run audit:editorial
```

Процесс добавления каждого урока:

1. Найти блок урока в source markdown.
2. Сверить с PDF, если OCR выглядит сомнительно.
3. Извлечь:
   - тему;
   - теорию;
   - лексику;
   - идиомы;
   - Technique 1;
   - Technique 2;
   - speech drilling;
   - ключи;
   - методические указания.
4. Прогнать editorial audit.
5. Исправить source artifacts и устаревшие формы.
6. Сформировать lesson object в `lib/course.ts` или будущей content-системе.
7. Добавить accepted alternatives только для естественных вариантов.
8. Проверить упражнения вручную.
9. Запустить:

```bash
npm run audit:editorial
npm run build
```

Текущий стек:

- Next.js App Router;

- API routes в `app/api/*`;
- TypeScript;
- Neon Postgres;
- Vercel Blob;
- OpenAI API;
- ElevenLabs API;
- deployment на Vercel.

Текущие ключевые файлы:

- `lib/course.ts` — структура и контент уроков;
- `lib/scoring.ts` — нормализация и локальная проверка;
- `lib/server/auth.ts` — auth/session;

- `app/page.tsx` — основной UI;

- `app/api/coach/check-answer/route.ts` — AI-проверка письменных ответов;

В будущем `lib/course.ts` желательно разбить на content modules:

- `content/course/lesson-01.ts`;

- `content/course/index.ts`;
- schemas in `lib/course-schema.ts`.

## 13. Progress model

Для каждого пользователя нужно хранить:

- текущий урок;
- текущий шаг;
- completed steps;
- best score by step;
- attempts count;
- слабые grammar points;
- слабые pronunciation points;
- need-review queue;
- last activity;
- audio recordings metadata.

Правило завершения шага:

- theory: пользователь нажал “Понял”;
- vocabulary: прослушал/повторил минимум требуемых элементов;
- technique: correct или acceptable score >= threshold;
- composition: минимум 10 валидных предложений;
- speaking: запись сделана, распознана, score >= threshold или consciously repeated;
- review: completed required repeats.

## 14. Quality criteria

Продукт считается коммерчески качественным, если:

- пользователь всегда понимает, что делать дальше;
- английский эталон скрыт там, где надо тренировать вспоминание;
- ошибки подсвечены там, где они возникли;
- AI объясняет кратко, ясно и по делу;
- нет редакторских ошибок из книги в primary content;
- TTS не генерируется повторно без необходимости;
- запись пользователя можно прослушать;
- speaking-модуль выглядит как единая тренировка, а не набор случайных карточек;
- desktop и mobile удобны;
- build проходит;
- основные flows не падают при отсутствии API quota, а показывают понятную ошибку.

## 15. MVP scope

Текущий MVP должен включать:

- login/password auth;
- admin user creation;
- Lesson 1 полностью;
- theory;
- vocabulary;
- idioms;
- Technique 1;
- 10-20 own sentences composition step;
- Technique 2;
- speaking drills;
- TTS via ElevenLabs;
- STT via OpenAI;
- pronunciation feedback;
- progress tracking;
- editorial audit pipeline.

## 16. Full product scope

Полная версия:

- все 44 урока;
- все speech drilling tracks regenerated or assembled via ElevenLabs;
- повторение предыдущих уроков по методике;
- dashboard слабых мест;
- spaced repetition;
- teacher/admin review;
- export progress;
- advanced pronunciation analytics;
- multi-user stable progress;
- RU/EN UI;
- mobile polished flow;
- offline-ish cached audio where possible.

## 17. Non-goals

На текущем этапе не нужно:

- платежи;
- email registration;
- social login;
- маркетинговый landing page вместо приложения;
- сложная LMS;
- сертификаты;
- open public signup.

## 18. Definition of Done for a lesson

Урок можно считать добавленным, если:

- есть теория;
- есть словарь и идиомы;
- все английские фразы проверены редакторски;
- все упражнения имеют target и accepted answers;
- spelling/grammar variants нормализованы;
- TTS работает и кешируется;
- speaking steps скрывают эталон;
- AI-проверка дает понятные объяснения;
- прогресс сохраняется;
- урок участвует в повторении будущих уроков;
- `npm run audit:editorial` не показывает block findings в `course-data` и `scoring`;
- `npm run build` проходит.

## 19. Главная продуктовая формула

Не “прочитал правило → сделал тест”, а:

> понял формулу → увидел русскую мысль → сам собрал английскую фразу → проверил → исправил → проговорил → сравнил с эталоном → повторил до автоматизма.

Именно эта формула должна определять все UX, архитектурные решения и приоритеты разработки.
